# Singularity — Windows next steps

Status right now: the Vulkan backend is **code-complete** (instance, device,
surface-via-callback, swapchain, compute + blit pipelines, HDR + bloom ping-
pong, per-frame descriptors and timestamps, screenshot capture). The
Windows CI leg compiles + unit-tests on every push; the `.msi` installer
is generated by GitHub Actions via CPack WIX. What's left is a pass on
actual Windows hardware to (a) run the renderer against a real GPU, (b)
verify the installer installs/uninstalls cleanly, and (c) decide the
code-signing posture.

## 1. Prereqs on a fresh Windows 10/11 box

```powershell
# 1. Visual Studio 2022 Community — install the "Desktop development with
#    C++" workload (MSVC, Windows SDK, CMake, git).
#    Or just Build Tools + CMake if you prefer the CLI.

# 2. Git + submodules-friendly clone:
git clone --recurse-submodules https://github.com/mal0ware/Singularity.git
cd Singularity

# 3. LunarG Vulkan SDK (provides dxc + the Vulkan runtime loader):
#    https://vulkan.lunarg.com/sdk/home  -> Windows installer (~400 MB)
#    Check VULKAN_SDK is on PATH after install:
echo %VULKAN_SDK%
where dxc
```

## 2. Build + run

```powershell
cmake -B build -G "Visual Studio 17 2022" -A x64 ^
    -DSINGULARITY_BACKEND_VULKAN=ON ^
    -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release -j

# Run:
.\build\app\Release\singularity.exe
```

You should see the same window the macOS Metal build produces: central
shadow, gravitational lensing of the accretion disc, ImGui panel on the
right for live controls. Dragging orbits the BH; scroll zooms; `Esc`
exits. The right-hand "Export -> Screenshot" button writes a timestamped
PNG to `%USERPROFILE%\Pictures\Singularity\`. Tunings you change through
the panel persist across runs via `%APPDATA%\Singularity\settings.conf`.

## 3. Runtime validation (Phase 4 / 5 gate)

```powershell
# Unit tests — physics only, no GPU needed. Should be green.
ctest --test-dir build -C Release --output-on-failure

# Python verification — includes the backend-equivalence test that
# perceptual-hashes Metal vs Vulkan golden images. On Windows without a
# matching Metal capture, these skip cleanly.
pip install -e verification
pytest verification/ -ra

# Interactive smoke tests:
.\build\app\Release\singularity.exe --res=1920x1080
.\build\app\Release\singularity.exe --kerr --spin=0.94
.\build\app\Release\singularity.exe --capture=schw.png --capture-ss=2
.\build\app\Release\singularity.exe --capture=kerr.png --kerr --spin=0.94

# Integrator perf sanity (JSON output, deterministic step count):
.\build\cli\Release\singularity_cli.exe --mode benchmark --metric schw
.\build\cli\Release\singularity_cli.exe --mode benchmark --metric kerr --spin 0.9
```

The `--capture=out.png` path renders one frame offscreen, reads it back
via `capture_frame()`, and writes a PNG. Diff it against the Mac Metal
golden in `verification/golden/metal/` using the ImageHash harness to
satisfy the cross-backend equivalence requirement in `docs/ARCHITECTURE.md
§12.3`. Perceptual-hash distance < 4 is the bar.

If the app fails at startup with "failed to load SPIR-V
geodesic_kernel.spv": the shader compilation step didn't run. Check that
`%VULKAN_SDK%\Bin\dxc.exe` was on PATH when you invoked `cmake -B build`
— `find_program(DXC_EXECUTABLE ...)` in `render/vulkan/CMakeLists.txt`
picks it up only at configure time.

## 4. Build the .msi installer

```powershell
# WiX 3.x. Already present on the windows-2022 GitHub runners; locally:
choco install wixtoolset -y
# Open a new shell so candle.exe + light.exe land on PATH.

cd build
cpack -G WIX -C Release
# -> Singularity-0.1.0-win64.msi in the build dir
```

The `.msi` lays out `singularity.exe` at `%ProgramFiles%\Singularity\bin\`
with the compiled SPIR-V under `bin\shaders\`, creates a Start-Menu
shortcut (`CPACK_WIX_PROGRAM_MENU_FOLDER = "Singularity"`), and wires
Programs-and-Features uninstall via the stable
`CPACK_WIX_UPGRADE_GUID` so future versions upgrade in place.

The same step runs in the `build-app` job of `.github/workflows/ci.yml`
and uploads the `.msi` as the `Singularity-windows-installer` artifact —
you can download it from any green CI run without needing WiX locally.

## 5. Code-signing decision

Windows SmartScreen flags unsigned executables with a "Microsoft Defender
SmartScreen prevented an unrecognized app from starting" dialog. Options
in rough priority order:

1. **Ship unsigned + document the workaround.** Users click "More info"
   -> "Run anyway". Cheap, but a turn-off for casual downloaders. This is
   the default posture for v0.7 and v1.0 per `docs/TODO.md`.
2. **OV cert (~$70-150/yr, e.g. DigiCert, SSL.com).** Removes the
   SmartScreen dialog after the cert "builds reputation" (a few hundred
   downloads). Not worth it unless the download volume is material.
3. **EV cert (~$300-500/yr).** Instant reputation, no warm-up. Only
   worth it if shipping to enterprise users.

To sign a `.msi` once a cert is in hand:

```powershell
# $env:SINGULARITY_SIGN_PFX is a Base64-encoded .pfx, drop-in from GHA
# encrypted secrets. Password via $env:SINGULARITY_SIGN_PASS.
[IO.File]::WriteAllBytes("cert.pfx",
    [Convert]::FromBase64String($env:SINGULARITY_SIGN_PFX))

signtool sign /f cert.pfx /p $env:SINGULARITY_SIGN_PASS ^
    /tr http://timestamp.digicert.com /td sha256 /fd sha256 ^
    build\Singularity-0.1.0-win64.msi
```

When you're ready, wire that into `.github/workflows/release.yml` on the
`windows-2022` job — same pattern the macOS job uses for Developer ID
signing via `SINGULARITY_MACOS_CODESIGN_IDENTITY`.
